# 从"聊天机器人"到"高权限智能体"：政策正在重新定义 AI 风险

> **发布时间：** 2026 年 5 月  
> **作者：** [@SangJieGe](https://github.com/SangJieGe/AI-Agent-Security-Research)  
> **适用平台：** GitHub · 微信公众号 · 知乎 · X (Twitter)

---

## 前言

2026 年 5 月 8 日，国家网信办、国家发展改革委、工业和信息化部联合印发了《智能体规范应用与创新发展实施意见》。

这是继大模型监管之后，国家层面第一次把"智能体"作为独立对象纳入系统性治理范围。

大多数人看到这份文件，会关注它开放了哪些应用场景、扶持了哪些产业方向。

但如果你是安全从业者，我建议你重点看另一面——**它在安全部分用了哪些词。**

这些词，正在精确描述一个工程现实：**AI 的风险重心，正在从"生成什么"转移到"执行什么"。**

---

## 第一部分：为什么这个政策值得安全行业重点关注

过去几年，AI 安全的讨论大多围绕一个核心问题展开：**模型会不会说错话、会不会被绕过、会不会泄露训练数据。**

这些是真实问题，但它们本质上是"文本问题"——AI 的输出是文字，危害路径是人读到错误内容后做出错误决策，中间还有一个人类判断的缓冲层。

而这份政策文件，第一次在官方语言里清晰描述了一种完全不同的对象。

文件对智能体的定义是：

> **"具备自主感知、记忆、决策、交互与执行能力的智能系统。"**

注意最后两个字：**执行。**

这不是在说一个生成文字的系统。这是在说一个**会采取行动的系统**——它可以调用工具、修改文件、发送请求、操作数据库、执行代码、触发部署流程。

这个定义的出现，意味着监管层已经清楚地意识到：**智能体不是升级版的聊天机器人，而是一个具备执行链路的自动化系统。**

当一个系统开始"执行"，它的安全问题就不再只是内容安全问题，而变成了**系统工程安全问题。**

---

## 第二部分：政策里最值得安全工程师逐字分析的几个词

文件在安全风险部分，官方给出了明确表述：

> **"智能体高自主性、高权限等特性也带来了隐私泄露、越权操作、行为失控等安全风险。"**

这三组词，不是官僚措辞，每一组都精准对应着真实的工程攻击面。

### 关键词一：高自主性

政策语言：智能体可以自主感知、决策、执行。  
工程现实：Agent workflow、autonomous code execution、CI/CD automation。

当一个 AI Agent 被赋予自主执行能力，它可能在没有人工确认的情况下：

- 自动修改代码并提交到主分支
- 自动触发部署流水线
- 自动调用外部 API 接口
- 自动生成并执行 Shell 脚本

自主性越高，人类的干预窗口越窄，错误传播的速度越快。

### 关键词二：高权限

政策语言：需要做好权限管理、行为管控。  
工程现实：GitHub token、cloud credentials、MCP tool access、runtime permissions。

今天的 AI Agent 为了完成任务，通常需要被授予相当高的权限：

```yaml
# 典型的 AI-driven GitHub Actions 配置
permissions:
  contents: write      # 可以修改代码
  pull-requests: write # 可以合并 PR
  packages: write      # 可以发布镜像
  secrets: read        # 可以读取 CI secrets
```

这个权限配置在"让 AI 帮我自动化开发流程"的需求下几乎是标配。但它意味着，一旦 Agent 被 prompt injection 攻击或工具链被污染，攻击者获得的不是一个文本输出，而是**一套对代码仓库、CI/CD 系统和生产环境的实际操作权限。**

文件中提出了一个非常重要的方向：**明确决策权限，厘清三类边界——仅限用户本人决策、需由用户授权决策、智能体自主决策。** 这其实就是工程上的"最小权限原则"在 Agent 场景下的政策化表达。

### 关键词三：行为失控

政策语言：防范越权操作、行为失控。  
工程现实：prompt injection、tool misuse、workflow hijacking、agent escalation。

"行为失控"在工程层面对应的是：Agent 在执行过程中，被外部输入（用户内容、工具返回、环境变量）中的恶意指令劫持，开始执行攻击者预设的操作，而不是用户授权的操作。

这是 MCP 协议攻击的核心机制，也是目前 Agent 安全研究中最活跃的攻击面。

### 关键词四：全周期安全管理

政策语言：实现智能体开发、部署、应用、维护全周期安全管理。  
工程现实：这正是 **AI DevSecOps** 的定义。

文件还专门提到了供应链安全：

> **"加强模型接入、应用程序接口调用、扩展工具使用等环节安全管理。"**

把这句话翻译成工程语言：MCP Server 的引入安全、第三方工具的权限边界、插件的供应链审计——这些都是当前 AI Agent 生态里还没有成熟解决方案的问题。

## 第三部分：真实工程场景里已经在发生什么

政策描述的风险不是假设场景，在真实的工程环境中，它们已经以具体的形态出现。

### 场景一：AI 自动修改 CI/CD 配置引入高权限

一个使用 AI Coding Agent 的开发团队，让 Agent 帮助优化 GitHub Actions 配置以提升自动化程度。Agent 生成了如下配置：

```yaml
on:
  push:
    branches: ["*"]   # 所有分支都触发

permissions: write-all  # AI 为了"确保不报权限错误"

jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      - run: ./deploy.sh  # AI 生成的自动部署脚本
```

每一行单独看都"能跑通"。但组合在一起：任何分支的 push 都可以触发拥有全量写权限的自动部署脚本。这是一个供应链污染的完美入口。

### 场景二：AI 生成的 Dockerfile 形成供应链漂移

```dockerfile
FROM ubuntu:latest          # 今天和明天的镜像可能不同
USER root                   # 高权限运行
COPY . .                    # 可能包含 .env、SSH key
RUN apt-get install -y curl && curl https://install.sh | bash
```

这个 Dockerfile 在自动化 DevOps 流程中被 Agent 生成、自动提交、自动构建、自动部署，全程无人 review。每一个步骤本身都是"AI 在帮忙"，但结果是把一个包含多个安全隐患的镜像推进了生产环境。

### 场景三：MCP 工具链的信任边界崩塌

一个典型的 AI Agent 在完成"帮我处理一个客户工单"的任务时，可能同时持有以下工具的访问权限：

```
GitHub MCP → 读写代码仓库
Slack MCP  → 发送消息、读取频道历史
Email MCP  → 发送邮件
Database MCP → 查询客户数据
```

当攻击者在任何一个输入源（客户工单、邮件内容、Slack 消息）中植入 prompt injection 指令，Agent 可能会在"处理工单"的名义下，实际执行跨系统的数据渗漏或权限提升操作。

这不是理论攻击。已有研究证实，Asana、Microsoft Copilot 等主流 MCP 集成中均存在可利用的真实安全漏洞。

## 第四部分：一个正在发生的范式转移

这是这篇文章最想说清楚的一件事。

**过去：AI 风险主要是"生成什么"的问题。**

模型输出有害内容、泄露隐私、产生幻觉——这些风险的危害路径是：AI 输出 → 人类读取 → 人类决策 → 产生影响。中间有人类判断这道防线。

**现在及未来：AI 风险正在变成"执行什么"的问题。**

Agent 调用工具、修改文件、触发部署、操作数据库——危害路径是：AI 决策 → 直接执行 → 产生影响。人类判断这道防线在自动化程度越来越高的工程环境中正在被绕过。

这个转变意味着：

- 传统的内容安全过滤器，无法防御工具调用层面的攻击
- 传统的代码安全审计，无法审计 AI 生成的 workflow 配置和 IaC 文件
- 传统的渗透测试方法，无法评估 Agent 在自主执行中的权限边界问题

**AI 的安全问题，正在从模型问题变成工程系统问题。**

这也是为什么这份政策文件值得安全行业认真对待——它不是在规范一个文字生成工具，而是在为一类具备执行能力的自动化系统建立治理框架。

## 第五部分：AI DevSecOps 可能会成为下一个结构性方向

基于以上分析，我认为以下方向在政策落地后会逐渐出现真实需求：

**Agent 权限治理（Agent Permission Governance）**  
针对 Agent 持有的工具权限、API 访问权限、文件系统权限进行细粒度管理和审计，实现政策中要求的"最小权限、可追溯、可撤销"。

**Workflow 安全审计（Workflow Security Review）**  
专门针对 AI 生成或 AI 辅助配置的 CI/CD Pipeline、GitHub Actions、Dockerfile、K8s YAML 等基础设施代码进行安全分析，识别权限过宽、供应链风险、密钥暴露等问题。

**Agent 运行时监控（Agent Runtime Security）**  
在 Agent 执行过程中实时监控工具调用行为，识别异常的跨工具访问、越权操作和可疑的执行链路，对应政策中"行为可验证、可追溯"的要求。

**AI 供应链安全扫描（AI Supply Chain Security）**  
针对 MCP Server、AI 插件、工具链组件的引入风险进行扫描和评估，对应政策中"供应链安全"和"扩展工具使用等环节安全管理"的要求。

**AI DevSecOps Baseline**  
为使用 AI Agent 进行开发的团队建立安全基线——从代码生成、IaC 生成、workflow 配置到自动化部署，每个环节应有什么安全门控，达到什么检测覆盖率。

这些方向目前还没有成熟的行业标准和工具体系。政策的落地会加速这个领域的需求形成。

---

## 结语

当智能体开始拥有权限、工具调用能力与执行链路，AI 安全的问题就不再只是模型问题，而会逐渐变成工程系统安全问题。

这份政策文件的意义，不只是规范了一个产业方向，更是第一次在官方语言层面，清晰地承认了这个转变的存在。

对安全从业者来说，这既是一个信号，也是一个窗口期。
